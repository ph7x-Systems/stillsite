# ADR-0035 — Two-factor authentication: TOTP, standard library, self-service

- **Status:** accepted
- **Date:** 2026-07-21

## Context

Panel accounts guard the editorial pipeline of a public site; a stolen
password should not be enough to publish. M7 schedules two-factor
authentication, and the project's rules apply: security changes carry an
ADR and hardening tests, no new dependencies without necessity, nothing
regresses (rate limiting, enumeration bounds, session model all stay).

## Decision

**TOTP (RFC 6238), the de-facto authenticator profile, standard library
only.**

- **Algorithm**: HMAC-SHA1, 30-second steps, 6 digits — what every
  authenticator app implements. `hmac`/`struct`/`base64` from the
  standard library; no new dependency.
- **Enrolment is self-service and confirmed**: a signed-in user opens
  the two-factor page, the server generates a 160-bit secret and shows
  it as a base32 key plus an `otpauth://` URI for manual entry (a QR
  image would need a new dependency; deliberately out until demand).
  Two-factor activates **only after the user echoes a valid current
  code**, proving the app holds the secret. CSRF-protected like every
  form.
- **Storage**: `users.totp_secret` (nullable) and `users.totp_step`
  (last accepted timestep) — one migration, all four engines, covered
  by the conformance suite. `totp_step` makes every code single-use:
  a replayed code inside its validity window is refused.
- **Sign-in**: the password step is unchanged — same limiter, same
  equalized work, same enumeration bounds; nothing about the account is
  revealed before valid credentials. When the account has two-factor
  enabled, valid credentials alone never create a session: the login
  re-renders asking for the code, and wrong codes spend the same
  rate-limit budget as wrong passwords. Verification accepts ±1 step of
  clock drift with constant-time comparison.
- **Disabling** requires a valid current code.
- **Recovery**: no backup codes in this phase (a recorded trade-off —
  they arrive with their own decision if demand appears). The recovery
  path is the one that already exists for lost passwords with lost
  email: `cms admin create-user --force` replaces the account — server
  access is the root of trust — which also clears two-factor state and
  revokes every session.

## Consequences

- Hardening tests grow: enrolment requires confirmation; sign-in with a
  wrong, replayed or missing code fails; the limiter covers code
  attempts; disable requires a code; accounts without two-factor sign
  in exactly as before.
- New panel strings join the i18n catalogs in the four languages
  (anti-drift enforced).
- The demo snapshot is unchanged (no account there has two-factor).
- Password reset (ADR-0032) intentionally does **not** clear two-factor
  state: mail access must not defeat the second factor.
